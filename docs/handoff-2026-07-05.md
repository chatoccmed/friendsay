## 3. กติกาที่ห้ามพลาด (ย่อจาก CLAUDE.md — อ่านตัวเต็มด้วย)

1. **Source of truth = repo `.github-ready`** (ตัวนี้) เท่านั้น · Windows ใช้ `npm.cmd` ไม่ใช่ `npm`
2. **`npm.cmd run build` ต้องผ่านก่อน commit ทุกครั้ง** · push `main` = deploy จริงทันที
3. **ห้าม force push เด็ดขาด** · commit เป็นชุดตามเรื่อง ข้อความชัดเจน
4. **Publish Gate 8 ข้อ** (บทความ roundup ขาดข้อเดียวห้าม commit): ภาพสินค้าจริงทุกรายการ / ตารางเทียบ + best-in-column / กราฟ ≥1 / quick-pick / จุดระวังจริงรายตัว / FAQ+schema / วันที่มองเห็น / ลิงก์ affiliate ทดสอบแล้ว — ราย­ละเอียดใน `design-upgrade-plan-2026-07.md`
5. **ระบบ B+D:** ห้ามเดายอดขาย ถ้า Shopee ไม่โชว์ให้เขียน "ไม่แสดง" ตรง ๆ ใช้จำนวนรีวิวแทน · ข้อมูลต้องมีวันที่เก็บกำกับ
6. **ไม่ใช้ภาษา AI/การตลาด/hype** · ห้ามอ้างว่าใช้เอง/ทดสอบเอง (ไม่มีสินค้าจริง) · ห้ามสร้างชื่อผู้ซื้อ/quote ปลอม
7. **สถานะร้านต้องตรงจริง** — official / authorized / เครือ / reseller อย่ารวบเป็น "official" ลอย ๆ (เจอบทเรียนนี้ในหน้าแอร์ต่ำกว่าหมื่น)
8. **ตรวจหน้า live ด้วยตา (screenshot/DOM) ก่อนรายงานว่าเสร็จ** ทุกครั้ง

## 4. วิธีทำงานที่พิสูจน์แล้วว่าได้ผล (สำคัญ — ทำตามนี้)

### 4.1 การทำบทความเปรียบเทียบ (spoke) ใหม่ — เร็วและปลอดภัย
บทความ spoke = ใช้ **ฐานข้อมูลแอร์ 10 รุ่นเดิม** re-frame เป็นธีมใหม่ (เช่น "แอร์ประหยัดไฟที่สุด", "แอร์ห้องนอนเงียบ") **ไม่ต้องเก็บ Shopee ใหม่** เพราะภาพ/ตัวเลข/ลิงก์มีครบใน `airConditioners.ts` แล้ว

ขั้นตอนที่ใช้ทำหน้า "แอร์ต่ำกว่าหมื่น" (ทำซ้ำได้):
1. เลือกเกณฑ์ธีม (เช่น priceNum < 10000, หรือ energyComfort สูงสุด) กรองรุ่นที่ผ่าน
2. **ให้ AI เขียนแค่ร้อยแก้ว** (intro/whoFor/จุดระวัง/FAQ/เลือกตามโจทย์) — **คน/โค้ดล็อกตัวเลข ลิงก์ ภาพ จาก data เอง** ห้ามให้ AI แต่งตัวเลข
3. ประกอบหน้าตามแม่แบบ Publish Gate: hero โทนครีม-พีช → quick-pick → การ์ดภาพใหญ่ → (ฟังก์ชัน: จัดเรียง/กรอง) → ตารางเทียบ best-in-column → กราฟ → เลือกตามโจทย์ → FAQ+schema → cross-link
4. **wire ≥3 ทางเข้า** (กันหน้า orphan): sitemap + footer (`SiteFooter.astro`) + หน้าแรก latest (`HomePage.astro`) + cross-link จากหน้า 10 อันดับ
5. `npm.cmd run build` → เปิด preview ตรวจด้วยตา → commit → push → curl ตรวจ live

> อ้างอิงโค้ดจริง: `src/pages/th/best/air-conditioners-under-10000.astro` เป็นแม่แบบที่ดีที่สุดของ spoke (มีฟังก์ชันจัดเรียง/กรองแบบ SSR-friendly reorder)

### 4.2 ตรวจคุณภาพแบบ 2 ชั้น (ใช้กับเนื้อหาสำคัญ)
1. **ร่าง → ขัดเกลา**: เขียนร่าง แล้วให้บรรณาธิการอีกคน/รอบ ตรวจว่าผูกข้อมูลจริง + ไม่มีภาษา AI
2. **ตรวจ adversarial**: ให้คนตรวจ 2 มุม (บรรณาธิการ/นโยบาย + เทคนิค/Publish Gate) หา finding แล้ว "ยืนยัน" แต่ละข้อว่าจริงก่อนแก้
   - รอบนี้จับได้ว่า badge "official" พูดเกินจริง (TCL=authorized, CANDY=ขายผ่านร้าน Haier) และพาดหัว "ตรวจของจริง" ขัดกับ FAQ ที่บอกว่าไม่มีสินค้าจริง → แก้แล้ว เป็นตัวอย่างว่าตรวจซ้ำมีค่า

### 4.3 การตรวจ live (สำคัญ)
- ใช้ preview tool ตรวจ DOM/ค่าสี/ฟังก์ชัน (แม่นกว่า screenshot) — **screenshot มักล้มเหลวถ้าหน้าต่าง Chrome ถูก minimize** ให้ตรวจผ่าน `preview_eval` แทน
- responsive ต้องผ่าน 390 / 820 / 1440 ไม่มี horizontal overflow
- หลัง push: `curl -s <url> | grep <marker>` วนจนเจอ marker แล้วตรวจซ้ำบนโดเมนจริง

## 7. หลักคิดของแบรนด์ (ถ้าลืมทุกอย่าง ให้จำอันนี้)

Friendsay ชนะด้วย **ความเชื่อใจ + เครื่องมือช่วยตัดสินใจ** ไม่ใช่การเป็นนักรีวิวที่มีของจริง (สู้ TikTok ไม่ได้)
ทุกอย่างวางอยู่บน **หลักฐานที่ตรวจย้อนได้ + ความซื่อสัตย์** — บทเรียนจาก Honey (โดนแฉเรื่องไม่โปร่งใสแล้วแบรนด์พังในสัปดาห์เดียว) คือเหตุผลว่าทำไมเราไม่เดาตัวเลข ไม่ขายอันดับ และบอกข้อเสีย/จุดระวังทุกรุ่น
funnel: **บทความ = ดึงคนเข้า (SEO/แชร์) · ช่องค้นหา = ใช้ซ้ำ · (อนาคต) LINE alert = ดึงกลับ**
